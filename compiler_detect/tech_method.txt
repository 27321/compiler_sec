2. 技术方案

2.1 总体方案概述

本发明提出一种基于多层次差异检测的深度学习模型编译安全评估方法。该方法的核心思路是：利用深度学习模型编译器在编译过程中产生的各阶段中间表示（Intermediate Representation，IR）及最终编译产物，通过对同一模型在不同抽象层次上的编译产物实施静态分析与差异性对比，系统性地检测编译链条中可能被注入的后门模式或恶意篡改行为，最终形成综合安全评估结论。

本发明不依赖于模型的实际执行，全程采用静态分析，不需要对模型进行推理。本发明亦不局限于特定编译器或特定深度学习框架，适用于任何能够在编译过程中输出各阶段中间表示的模型编译系统。

如图2所示，本发明方法按顺序包括以下五个步骤：
（1）使用编译器对深度学习模型进行编译，生成各阶段中间表示及最终编译产物；
（2）对高层计算图IR文件进行单文件静态检测，检测是否存在后门模式；
（3）对融合前IR文件与融合后IR文件进行差异性分析，验证其是否符合正常编译流程变化；
（4）对Triton内核源码文件与机器码反汇编文件进行差异性分析，验证其是否符合正常编译流程变化；
（5）根据上述多层次结构检测结果，对模型编译进行综合安全评估。

---

2.2 步骤一：编译器调用与中间产物生成

如图1所示，将待检测的深度学习模型输入编译器，触发完整编译流程，并在各编译阶段保留所有中间产物。以PyTorch编译器为例，具体包括以下阶段及其产物：

（1）图捕获阶段（Graph Capture）：编译器通过字节码分析捕获模型的Python级动态计算图，生成调试日志文件（debug.log）。

（2）高层图IR生成阶段（High-Level Graph IR）：编译器将捕获的计算图转换为算子级展开的高层图中间表示，生成以下文件：
    a. fx_graph_readable.py：可读性FX图，每个节点以低层算子（aten算子）序列表示，并附有源码注释，适合静态分析；
    b. fx_graph_runnable.py：可运行FX图，与fx_graph_readable.py在语义上等价，但格式可直接执行；

（3）图变换与分解阶段（FX Graph Transform & Decomposition）：编译器对高层图进行若干优化变换，生成：
    c. fx_graph_transformed.py：经过Inductor变换后的图IR；

（4）底层IR生成阶段（Inductor Lowering）：编译器将高层图IR降层为调度器IR，生成：
    d. ir_pre_fusion.txt：算子融合优化前的调度器IR；

（5）IR优化阶段（Inductor IR Optimization）：编译器执行算子融合等优化，生成：
    e. ir_post_fusion.txt：算子融合优化后的调度器IR；

（6）代码生成阶段（Code Generation）：编译器将优化后的IR生成可执行内核代码，生成：
    f. output_code.py：面向GPU平台时包含Triton内核调用的Python宿主代码；
    g. 各Triton内核的源码文件（*.source，即TTIR格式的Triton中间表示）；
    h. 各Triton内核的编译二进制文件（*.cubin），对其执行反汇编操作，得到SASS格式的GPU汇编文件（*.sass）。

上述产物中，fx_graph_readable.py、fx_graph_runnable.py、fx_graph_transformed.py和output_code.py作为单文件静态检测的对象；ir_pre_fusion.txt与ir_post_fusion.txt作为融合前后差异性分析的对象；*.source与*.sass文件作为内核层差异性分析的对象。

2.3 步骤二：高层计算图IR单文件静态检测

本步骤对fx_graph_readable.py（以及fx_graph_runnable.py、fx_graph_transformed.py、output_code.py，以下以fx_graph_readable.py为主要说明对象）进行单文件静态检测，目的是识别高层计算图中以数学计算结构形式嵌入的神经网络后门模式。如图3所示，具体检测流程包括以下子步骤：

2.3.1 FX图解析与算子提取

读取FX图文件，利用正则表达式逐行解析，提取计算图中所有算子节点及其数据依赖关系，形成统一的"算子-变量"数据流表示。具体地：
- 对每条形如"变量名 = torch.ops.aten.算子名(参数列表)"的节点行，提取变量名、算子名及参数列表，构建aten算子映射表aten_ops，格式为：{变量名: (算子名, [参数列表])}；
- 对每条形如"变量名 = torch.ops.prims.算子名(参数列表)"的节点行，提取对应信息，构建prims算子映射表prims_ops，格式与aten_ops相同；
- 参数列表解析时，采用括号深度跟踪的方式正确处理嵌套参数结构，确保逗号分割的准确性。

2.3.2 辅助分析结构构建

在执行后门检测前，构建以下三类辅助分析结构：

（a）变量别名映射（alias_map）构建：

FX图中存在一类"透明传递算子"，其输出在数值上等价于第一个输入，仅改变形状或数据类型，不改变数值语义，包括：clone、view、reshape、expand、expand_as、permute、unsqueeze、squeeze、contiguous、alias、detach、t等aten算子，以及prims中的convert_element_type算子（类型转换）。

对上述透明传递算子，建立变量别名映射：alias_map[新变量名] = 源变量名。进一步，对存在链式传递的情形（如变量a经view得到b，b经clone得到c），执行路径压缩，将所有链式别名统一解析为最终规范变量名（canonical variable），即alias_map[c] = a。

【关键改进点】：传统的后门检测方法将经过形状变换的变量视为独立变量，导致同一张量经过unsqueeze、expand等操作后无法被识别为同一掩码变量。本发明引入透明算子别名追踪机制，使掩码变量在经过任意数量的透明传递操作后仍能被一致识别，显著降低漏报率。

（b）全1张量集合（ones_set）识别：

扫描所有aten算子，识别产生全1张量的变量，包括：由ones.*、ones_like.*算子产生的变量；由full.*、full_like.*算子产生且填充值为1.0的变量。在别名映射构建完成后，将所有指向上述全1张量变量的别名变量也纳入全1张量集合。

（c）激活门控变量映射（gate_map）构建：

扫描所有aten算子，识别输出值域受限的激活函数的输出变量，包括：sigmoid（输出值域[0,1]）、tanh（输出值域[-1,1]）、hardsigmoid、hardswish等算子的输出变量，构建软门控变量映射gate_map，格式为：{门控变量名: 规范化输入变量名}。这类变量可被攻击者用作软触发权重，以替代硬掩码实现更隐蔽的后门。

2.3.3 多模式后门检测

基于上述辅助结构，并行执行以下七类后门检测器：

【检测器A】标准Blending后门检测（高置信度）：

检测以下数学形式的标准混合型后门：
    result = mask * malicious_output + (1 - mask) * normal_output

该结构在FX图算子级别展开为四步数据流：
    step1: sub_var = sub(1.0, mask) 或 sub(ones_tensor, mask)    // 计算 (1 - mask)
    step2: mul_a   = mul(mask, malicious)                         // 计算 mask * A
    step3: mul_b   = mul(sub_var, normal)                         // 计算 (1-mask) * B
    step4: result  = add(mul_a, mul_b)                            // 合并两路

检测逻辑：
- 第一步，扫描所有sub算子，识别"sub(1.0, 变量)"或"sub(全1张量, 变量)"形式的操作，建立(1-mask)计算映射；同时识别rsub.Scalar(mask, 1.0)（等价于1.0-mask）；
- 第二步，扫描所有mul算子，找到以上述(1-mask)结果为因子的乘法，建立(1-mask)*B的乘法映射；
- 第三步，扫描所有mul算子，找到以mask变量（含别名穿透）为因子的乘法，建立mask*A的乘法映射；
- 第四步，扫描所有add算子，找到同时以mask*A结果和(1-mask)*B结果为加法因子的加法操作，即完成四步数据流闭合检测。

同时检测lerp算子（等价形式B）：lerp(start, end, weight) ≡ (1-weight)*start + weight*end，直接等价于标准Blending结构，将三参数均为变量的lerp操作标记为高置信度发现。

【检测器C】Where条件选择型后门检测（中置信度）：

检测以下数学形式的条件选择型后门：
    result = where(trigger_condition, malicious_output, normal_output)

扫描所有where算子，对条件参数、真值参数、假值参数均为变量名（而非常量）的情形上报，以排除合法的常量条件分支。

【检测器D】等价Blending变形检测（低置信度）：

检测以下数学变形形式的混合后门：
    result = normal + mask * (malicious - normal)
           ≡ mask * malicious + (1 - mask) * normal

该形式省去了显式的(1-mask)计算，在FX图中展开为三步数据流链：
    diff   = sub(malicious, normal)
    delta  = mul(mask, diff)
    result = add(normal, delta)

检测逻辑：首先收集所有sub(A, B)（排除sub(1.0, X)形式以避免与检测器A重叠），建立差值映射；再找mul(mask, diff)的乘法；最后确认add(B, delta)中的B与diff中的被减数为同一逻辑张量（穿透别名）。

【检测器E】加性触发注入检测（中置信度）：

检测以下数学形式的加性注入型后门：
    result = normal_output + trigger * delta_vector

该形式无互补项(1-trigger)*normal，攻击者使trigger变量在正常输入下趋近于0，在触发输入下激活，直接将扰动量叠加到正常输出。在FX图中展开为两步数据流链：
    perturbation = mul(trigger, delta_vector)
    result       = add(normal_output, perturbation)

检测条件：mul的两个因子均须为中间计算变量（非原始模型参数）；add的基底变量亦须为中间变量；排除已被检测器A和D覆盖的结果变量，避免重复上报。

原始模型参数的识别：FX图中，模型权重和偏置以argN_M（如arg0_1、arg3_1等）形式注入，通过正则匹配可与运行时中间计算变量区分，从而降低将合法LayerNorm归一化（mul(activation, weight_argN)）误报为注入操作的概率。

【检测器F】激活函数门控型后门检测（低置信度）：

检测以软激活函数门控替代硬掩码的后门变体：
    gate   = sigmoid(trigger_feature)    // 软门控，值域[0,1]
    result = gate * malicious + base     // 无显式(1-gate)项

检测逻辑：以gate_map中的激活门控变量为基础，找mul(gate_var, other)的门控乘法，再找add(base, gate_mul_result)的叠加操作。

【检测器G】源路径异常检测（信息性，中/低置信度）：

FX图的每个节点附有源码位置注释，格式为：
    # File: /path/to/file.py:行号 in 方法名

扫描所有此类注释，识别路径不属于已知标准框架目录（transformers、torch、numpy、scipy、sklearn、huggingface_hub等）的来源路径，这类路径可能指向攻击者插入的自定义恶意代码模块。同时统计"No stacktrace found"节点数量，异常过多时作为辅助线索。

2.3.4 置信度分级与安全判断

将所有检测器的发现按置信度分为三级：
- 高置信度：精确匹配后门数学特征，误报率极低，包括标准Blending和Lerp等价形式；
- 中置信度：结构可疑，但存在合法场景，包括Where条件选择、加性触发注入、未知源路径；
- 低置信度：广泛存在于合法模型中仅供参考，包括等价Blending变形、激活函数门控、无栈追踪节点。

综合判定规则如下：
- 存在结构型高置信度发现 → 判定为BACKDOOR DETECTED（需立即人工审查）；
- 存在结构型中置信度发现 → 判定为BACKDOOR DETECTED（建议人工审查）；
- 仅存在源路径中置信度发现 → 判定为SOURCE ANOMALY（核查模型来源）；
- 仅存在低置信度发现 → 判定为SUSPICIOUS（建议进一步审查）；
- 无任何发现 → 判定为SAFE（未发现可疑结构）。

2.4 步骤三：融合前后调度器IR差异性分析

本步骤对ir_pre_fusion.txt（算子融合前调度器IR）与ir_post_fusion.txt（算子融合后调度器IR）进行差异性分析，目的是验证编译器算子融合优化过程所引入的结构变化是否符合正常编译行为，从而识别在融合阶段注入的异常算子或数据流。如图4所示，具体流程如下：

2.4.1 调度器IR结构解析

读取ir_pre_fusion.txt和ir_post_fusion.txt文件内容，通过正则表达式进行结构解析，提取计算图中的关键元素：

（a）算子集合提取：识别所有形如"算子名: 算子类型"的调度节点，包括：
    - SchedulerNode（普通计算节点）；
    - FusedSchedulerNode（融合后的复合节点）；
    - ExternKernelSchedulerNode（外部内核调度节点，对应调用外部计算库）；

（b）缓冲区集合提取：识别所有以"buf数字"格式命名的中间缓冲区变量；

（c）外部内核映射提取：识别形如"算子名.node.kernel = 外部内核名"的外部内核绑定关系。

2.4.2 融合前后结构统计信息计算

分别统计融合前（PRE）和融合后（POST）调度器IR中各类算子的数量分布（SchedulerNode数量、FusedSchedulerNode数量、ExternKernelSchedulerNode数量），及总算子数量和总缓冲区数量，分析融合优化带来的结构变化规模。

2.4.3 算子集合差异比较与异常算子识别

对融合前后IR的算子集合执行集合差运算：
- 仅在PRE中存在的算子：通常对应被融合消除或重排的算子，属于正常融合行为；
- 仅在POST中存在的算子：需进一步分析其类型：
    - 若为FusedSchedulerNode，属于正常的融合产物；
    - 若为SchedulerNode或ExternKernelSchedulerNode，则为异常：这类算子在融合前完全不存在，融合后凭空出现且非融合节点，需人工确认是否为合理新增算子或被注入的恶意计算节点。

【关键改进点】：正常的算子融合操作只会将多个SchedulerNode合并为FusedSchedulerNode，不会无中生有地增加独立SchedulerNode或ExternKernelSchedulerNode。本发明利用此编译不变量，将仅在POST中存在的非FusedSchedulerNode类型算子标记为异常，实现对融合阶段注入操作的检测。

2.4.4 缓冲区结构差异分析

计算融合前后中间缓冲区集合的差异：
- 仅在PRE中存在的缓冲区：通常为被融合内化的临时缓冲区，属于正常行为；
- 仅在POST中存在的缓冲区：需关注其数量；
- 若POST中缓冲区总数量超过PRE中缓冲区总数量的1.5倍，则触发异常告警，提示可能存在额外的数据流分支或隐藏的中间计算路径。

2.4.5 外部计算内核差异分析

对融合前后IR中的外部内核名称集合执行集合差运算：
- 仅在POST中出现的外部内核名称（新增外部内核）：需重点人工审查，确认是否为预期优化新增的算子调用，或被注入的非预期外部计算；
- 仅在PRE中出现的外部内核名称（消失的外部内核）：通常为被融合替代，属正常行为，可选择性关注。

2.4.6 综合差异结论

综合算子、缓冲区和外部内核的差异检测结果，生成结构化分析报告，识别可能存在的异常编译行为。报告内容包括：总体统计信息、各类差异项列表及异常标记、需人工核查的具体节点列表。

2.5 步骤四：Triton内核Source-SASS差异性分析

本步骤对各Triton计算内核的*.source文件（Triton中间表示，即TTIR格式）与对应*.cubin文件反汇编得到的*.sass文件（SASS格式GPU汇编代码）进行差异性一致性分析，目的是验证机器码层面的计算操作是否与源码层面的计算语义保持一致，从而检测在代码生成或二进制编译阶段注入的恶意篡改。如图5所示，具体流程如下：

2.5.1 源码文件与机器码文件解析

（a）*.source文件解析：读取Triton内核的TTIR源码文件，提取以下信息：
    - 算子集合（ops）：所有出现的TTIR操作符，格式为"命名空间.操作名"（如tt.load、tt.store、arith.addf、arith.andi等），统计每个操作符的出现次数（op_count）；
    - 整数常量集合（constants_int）：提取arith.constant指令、dense<N>字面量及裸整数注释中的所有整数常量；
    - 浮点常量集合（constants_float）：提取arith.constant指令及dense<N.NNN>字面量中的浮点常量；
    - 整数位运算标志（has_data_bitops）：标记源码是否存在arith.andi、arith.ori、arith.xori等整数位操作；
    - 可疑关键词命中列表：在注释文本和字符串字面量中扫描预定义的可疑关键词（包括backdoor、inject、payload、shellcode、exploit、malware、trojan、rootkit、bypass、hook、poison等），元数据行（#loc定义行）在扫描前预先过滤，避免因文件路径名中的关键词触发误报。

（b）*.sass文件解析：读取SASS格式反汇编文件，仅解析其中的.text代码节区（跳过.debug_frame、.debug_line等调试节区以减少误报）。对代码节区中的每条指令，提取：
    - 指令偏移（offset）：来自/*十六进制偏移*/格式；
    - 谓词（predicate）：如@P0、@!P2等条件执行谓词（可选）；
    - 助记符（mnemonic）：指令操作码，如LOP3、STG、FADD、FMUL、MOV等；
    - 操作数列表（operands）：逗号分隔的寄存器、立即数等操作数；
    - 十六进制立即数集合：从操作数中提取所有形如0x十六进制数的立即数常量；
    - 十进制浮点字面量集合：从操作数中提取形如数字.数字格式的浮点立即数。

2.5.2 可解释常量集合构建

【关键改进点】：本发明提出"可解释常量集合"概念，即基于源码层面的常量信息推导出机器码层面合法出现的常量范围，用于区分编译器正常行为产生的机器码常量与注入的非法常量。可解释常量集合构建规则如下：

（a）整数可解释集合（explainable_int）构建：
    - 对源码中每个整数常量c，将c与{1, 2, 4, 8, 16, 32, 64, 128, 256}的乘积加入集合，覆盖编译器为向量化和内存对齐目的对常量进行缩放的情形；
    - 对源码中每个浮点常量f，将其IEEE-754单精度位模式（即以4字节整数形式表示的浮点数编码）加入集合，覆盖编译器将浮点常量以整数形式嵌入逻辑运算指令的情形；
    - 加入常见GPU编译器内置常量，包括：常见整数溢出边界（0x7fffffff、0x80000000、0xffffffff）、IEEE-754特殊值（正负无穷大0x7f800000/0xff800000）、常用浮点值的位模式（1.0f对应0x3f800000、2.0f对应0x40000000、0.5f对应0x3f000000）；
    - 加入常见线程/地址位掩码（0x1f、0x3f、0x7f、0xff、0xfff、0xffff等），覆盖GPU线程束（warp）索引计算的正常需求；
    - 仅将绝对值大于0x1000（即4096）的常量纳入审查范围，小于此阈值的常量属于常见立即数，不作为告警依据。

（b）浮点可解释集合（explainable_float）构建：
    - 加入源码中所有浮点常量及其负数；
    - 加入源码中整数常量的浮点形式及其倒数和负倒数（覆盖divf N → FMUL 1/N等编译优化转换）；
    - 加入常见GPU编译器浮点常量（0.0、1.0、-1.0、0.5、-0.5、2.0等）及2的幂次常量。

2.5.3 多规则一致性检测

基于以上解析结果和可解释常量集合，执行以下八条检测规则：

【规则1】操作符一致性检测（OP_CONSISTENCY，严重级别：CRITICAL）：

扫描SASS代码中所有逻辑运算指令（助记符包含LOP3、AND、OR、XOR的指令），若某条逻辑运算指令携带的十六进制立即数大于0x1000且不在可解释整数集合中，则触发CRITICAL告警。此规则的判定依据为：SASS中出现的逻辑运算常量若无法从源码常量推导，极可能为注入的位掩码（用于通过位或操作将攻击者预设的值叠加到计算结果上）。

【规则2】写入前异常计算序列检测（PRESTORE_SEQUENCE，严重级别：CRITICAL）：

以滑动窗口方式（默认窗口大小为8条指令）遍历SASS指令序列，以每条全局存储指令（STG、STL、STS等）为终点，向前检查窗口内是否同时存在以下三类指令且满足浮点计算→位运算→存储的顺序：
    - 浮点计算指令（FADD、FMUL、FFMA、DADD、DMUL、DFMA、MUFU等）；
    - 逻辑位运算指令（LOP3、AND、OR、XOR）且携带不可解释的大立即数常量；
    - 全局存储指令。

若满足上述条件，触发CRITICAL告警。该序列模式对应"正常浮点计算→结果被篡改→写回内存"的数据污染特征。

【规则3】条件位运算检测（CONDITIONAL_DATA_OP，严重级别：CRITICAL）：

扫描所有带有谓词（predicate字段非空）的逻辑运算指令（LOP3、AND、OR、XOR），若该指令携带的十六进制立即数大于0x1000且不在可解释整数集合中，则触发CRITICAL告警。谓词保护的逻辑运算意味着攻击者可能只对满足特定条件（如触发器激活）的线程执行篡改操作，具有高度针对性。

【规则4】常量来源审计（CONSTANT_PROVENANCE，严重级别：HIGH）：

扫描SASS中所有逻辑运算指令携带的十六进制常量，对每个大于0x1000且不在可解释整数集合中的常量，触发HIGH告警，报告该常量值及所在指令。此规则是规则1的补充，对重复出现的相同常量仅上报一次（去重处理）。

【规则5】可疑关键词扫描（KEYWORD_SCAN，严重级别：MEDIUM）：

对*.source文件的过滤后内容（已移除元数据行）和*.sass文件的注释行，扫描预定义可疑关键词。在源文件中，关键词扫描仅针对注释文本（//之后的内容）和字符串字面量。"ub.poison"被列为白名单（MLIR UB方言的合法标记），不触发告警。

【规则6】浮点常量注入检测（FP_CONST_INJECTION，严重级别：HIGH）：

以滑动窗口方式（默认窗口大小为8条指令），在全局存储指令前的窗口内，扫描浮点计算指令（FADD、FMUL、FFMA）携带的十进制浮点字面量。若某浮点字面量无法在可解释浮点集合中找到（采用相对误差1e-5的容差比较），则触发HIGH告警。此规则覆盖攻击者通过FP算术指令（而非位运算）注入偏置的变体，例如"FMUL R9, R9, 0.888888"这类通过浮点乘法修改计算结果的攻击形式。

【规则7】写入前广义常量覆盖检测（BROAD_PRESTORE_CONST，严重级别：HIGH）：

以较短的滑动窗口（默认窗口大小为3条指令）扫描每条全局存储指令前的窗口，对窗口内所有非逻辑运算指令（排除已由规则1-4覆盖的LOP3类）且携带大于0x1000的不可解释立即数常量的指令触发HIGH告警。此规则覆盖通过MOV（寄存器直接赋值）、IADD3（整数加法）、IMAD（整数乘法）等指令写入攻击常量的变体。

【规则8】存储操作数量异常检测（STORE_COUNT_ANOMALY，严重级别：HIGH/MEDIUM）：

统计*.source文件中tt.store指令的出现次数（source_stores）和*.sass文件中全局存储指令（STG等，排除原子操作ATOM和规约操作RED）的出现次数（sass_stores）。计算允许上限：expected_max = source_stores × 6 + 2，其中系数6为Triton向量展开的最大倍率上界，+2为常量缓冲写入等额外开销。若sass_stores超过expected_max，则触发告警（超出量≤5条时为HIGH，超出量>5条时建议提升为CRITICAL）；若source_stores为0而sass_stores大于0，则直接触发HIGH告警。注入的额外全局存储操作可能对应数据泄露（将计算中间结果或模型参数写入攻击者控制的内存地址）。

2.5.4 检测结果去重与置信度评分

对所有规则触发的检测发现，以（规则名称、证据字符串）为复合键进行去重，避免同一SASS指令被多条规则重复上报导致虚高评分。

按规则严重级别对发现进行分组（CRITICAL、HIGH、MEDIUM、LOW），计算置信度评分：若存在任意CRITICAL或HIGH级别发现，判定为INFECTED（感染，置信度随CRITICAL发现数量递增）；若仅有MEDIUM/LOW级别发现，判定为SUSPICIOUS（可疑）；若无任何发现，判定为CLEAN（清洁）。

2.6 步骤五：综合安全评估

综合上述三个检测模块（高层图IR单文件静态检测、融合前后IR差异性分析、Source-SASS差异性分析）的检测结果，对模型编译进行最终安全评估，生成结构化评估报告。

综合评估采用以下判定逻辑：
（1）若高层图IR单文件静态检测发现高置信度或中置信度的结构型后门模式，判定为存在高风险后门，需立即人工审查；
（2）若融合前后IR差异性分析发现仅在POST中存在的非融合类异常算子，或缓冲区数量异常增长，或新增外部内核，判定为编译过程存在可疑结构变化，需核查融合阶段；
（3）若Source-SASS差异性分析发现CRITICAL或HIGH级别告警，判定为机器码层面存在注入行为，需检查代码生成阶段；
（4）若上述三层检测均无异常发现，判定模型编译过程为安全。

综合安全评估报告包含以下内容：各层次检测模块的详细发现列表（含置信度/严重级别、触发的具体证据）、各层次模块的单独判定结论、综合安全等级（安全/可疑/危险）及建议的后续处置措施（人工审查范围、建议核查的具体文件与代码位置）。

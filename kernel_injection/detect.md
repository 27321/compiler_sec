# 第X章 基于编译产物一致性的Triton内核注入检测方法

## X.1 方法概述

Triton编译器将用户编写的Python内核经由TTIR、LLVM IR、PTX等中间形式最终编译为在GPU上执行的SASS机器码。在正常编译过程中，SASS中出现的所有操作语义和立即数常量均可在源文件中找到对应的计算逻辑——操作类型由源文件中的算子决定，常量值由源文件中显式声明的字面量或其经由指针缩放、类型转换等合法变换的衍生值决定。这种来源上的可追溯性构成了本章检测方法的核心依据。

当攻击者在PTX生成阶段植入恶意指令时，被注入的操作及其携带的常量在源文件中没有对应的来源。以第X章描述的参考攻击为例，注入指令`or.b32 %r4, %r4, %backdoor_marker_b32`在SASS中表现为`LOP3.LUT R9, R9, 0x3fe1c71c, RZ, 0xfc, !PT`，其中常量`0x3FE1C71C`（IEEE-754单精度浮点数0.888...的位模式）在源文件中既未作为整数常量出现，也无法通过对源文件中任何已有常量施加合法的编译器变换得到。这种不可解释性是植入攻击区别于正常编译行为的根本特征。

基于上述观察，本章提出一种静态的跨产物一致性检测方法。该方法分为两个阶段：第一阶段分别解析TTIR源文件和SASS反汇编，从中提取结构化的语义模型，并依据源文件信息构建"可解释常量集合"作为判断基准；第二阶段将若干针对不同攻击维度的检测规则依次应用于SASS模型，凡SASS中出现无法由可解释集合覆盖的常量、无法由源文件操作类型解释的逻辑运算，或具有特征性时序结构的指令序列，均被标记为异常发现项，最终汇总生成检测报告。该方法无需掌握攻击的具体实现方式，可对多种注入变体进行统一检测。

## X.2 编译产物的解析与建模

### X.2.1 Triton IR源文件解析

TTIR是基于MLIR框架的Triton专用方言，其基本结构为若干嵌套的`func.func`模块，每个模块对应一个GPU内核。本文从TTIR源文件中提取以下信息用于后续分析。

**操作符集合**。遍历源文件中所有MLIR操作名称（格式为`dialect.opname`，如`arith.addf`、`tt.store`），构建操作符集合及各操作符的出现次数。该信息用于判断SASS中出现的操作类型是否有对应的源文件依据，以及统计`tt.store`的出现次数作为存储规模基准。

**常量集合**。提取源文件中所有显式声明的字面量，包括整数常量（来自`arith.constant N : iM`、`dense<N>`等形式）和浮点常量（来自`arith.constant N.N : fM`、`dense<N.N>`等形式）。这些常量是构建可解释常量集合的原始输入。

**元数据行过滤**。TTIR文件中包含大量调试位置注解，格式为`#locN = loc("path/to/file":line:col)`。这类行中的路径字符串可能含有与安全研究相关的词汇（如用于存放检测代码的目录名），若不加过滤则会在关键词扫描阶段产生误报。本文以正则表达式`r'#loc\d*\s*=\s*loc\s*\('`识别并在预处理阶段剔除全部元数据行，关键词扫描仅针对过滤后的有效代码行进行。

### X.2.2 SASS目标文件解析

`cuobjdump --dump-sass`生成的SASS反汇编文件包含多个节区（section），其中`.text`节区保存可执行指令序列，`.debug_frame`、`.debug_line`、`.nv_fatbin`等节区保存调试信息和元数据。调试节区中包含大量十六进制数值，若不加区分地纳入分析范围，将对常量检测规则造成严重干扰。本文的解析器以`.section .text`前缀标题行为起点开始收集指令，遇到下一个非`.text`类型的`.section`标题行时停止，仅对`.text`节区内的指令执行后续分析。

SASS指令在反汇编文件中的标准格式为：

```
        /*OFFSET*/    [@PREDICATE]    MNEMONIC[.MODIFIER]    OPERANDS ;
```

以本文研究的注入指令为例：

```
/*0440*/        @P0       LOP3.LUT    R9, R9, 0x3fe1c71c, RZ, 0xfc, !PT ;
```

本文将每条指令解析为包含以下字段的结构体：指令偏移（`offset`，十六进制整数）、谓词（`predicate`，如`@P0`、`@!P2`，无谓词时为空）、助记符（`mnemonic`，如`LOP3`，统一转换为大写）、操作数列表（`operands`，以逗号分隔后去除空白）。在操作数列表上进一步提取两类数值：以`0x`开头的十六进制立即数，以及格式为`[-]d.ddd`的十进制浮点字面量，分别用于整数常量规则和浮点常量规则的分析。

### X.2.3 可解释常量集合的构建

可解释常量集合（Explainable Constant Set，以下记为$\mathcal{E}$）是本章方法的核心抽象，其作用是将"合法的编译器行为所能产生的常量"系统性地枚举出来，作为后续异常判断的参照基准。SASS中出现的常量若属于$\mathcal{E}$，则视为可由源文件解释的合法值；若不属于$\mathcal{E}$，则视为注入的可疑值。

整数可解释集合的构建来自以下四个来源。第一，对源文件中每个整数常量$c$，将$c \times 2^k$（$k = 0, 1, \ldots, 8$）加入$\mathcal{E}$，覆盖编译器在向量化、内存对齐和指针步长计算中对常量进行2的幂次倍缩放的情形。第二，对源文件中每个浮点常量$f$，计算其IEEE-754单精度位模式并加入$\mathcal{E}$，覆盖编译器将浮点字面量以整数形式嵌入指令操作数的情形。第三，无条件加入若干编译器内置常量，包括：GPU执行模型所依赖的掩码值（warp内线程编号掩码`0x1f`、字节掩码`0xff`等），以及IEEE-754特殊值的位模式（正无穷`0x7F800000`、负无穷`0xFF800000`）和常用浮点常量的位模式（1.0f对应`0x3F800000`、0.5f对应`0x3F000000`等）。第四，值小于等于`0x1000`的常量不纳入$\mathcal{E}$的检查范围，因为该范围内的小整数和小偏移量在任何场景下均属正常，无足够的区分度作为注入特征。

浮点可解释集合$\mathcal{E}_f$在整数集合的基础上单独构建，用于浮点字面量的验证。对源文件中每个整数常量$c$和浮点常量$f$，将$c$、$f$、$-c$、$-f$，以及$1/c$和$1/f$（零值除外）加入$\mathcal{E}_f$，覆盖编译器在实现除法时将除数转化为乘以倒数的常见变换。此外无条件加入$\{0.0, \pm0.5, \pm1.0, \pm2.0, 0.25, 0.125, 0.0625\}$等常见编译器浮点常量。

## X.3 检测规则

本章共设计八条检测规则，在同一检测框架下从不同维度验证SASS与源文件之间的语义一致性。八条规则依次执行，各自独立产生发现项（Finding），每个发现项记录规则名称、严重等级和具体证据字符串，最终经去重后汇总输出。规则与严重等级的对应关系如表X-1所示。

**表X-1 检测规则汇总**

| 规则 | 名称 | 严重等级 |
|------|------|----------|
| R1 | OP\_CONSISTENCY | CRITICAL |
| R2 | PRESTORE\_SEQUENCE | CRITICAL |
| R3 | CONDITIONAL\_DATA\_OP | CRITICAL |
| R4 | CONSTANT\_PROVENANCE | HIGH |
| R5 | KEYWORD\_SCAN | MEDIUM |
| R6 | FP\_CONST\_INJECTION | HIGH |
| R7 | BROAD\_PRESTORE\_CONST | HIGH |
| R8 | STORE\_COUNT\_ANOMALY | HIGH |

### X.3.1 逻辑操作常量来源验证（R1、R4）

R1（OP\_CONSISTENCY）和R4（CONSTANT\_PROVENANCE）从常量来源可追溯性的角度验证SASS中逻辑运算指令的合法性。二者均遍历SASS`.text`节区中助记符属于逻辑/移位运算类（LOP3、AND、OR、XOR、SHF等）的全部指令，对每条指令的操作数中提取到的十六进制立即数$c$，判断$c$是否满足$c > \texttt{0x1000}$且$c \notin \mathcal{E}$。若满足，则说明该常量无法由源文件中的任何信息合法推导，被标记为可疑。

R1与R4的区别在于严重等级：R1面向CRITICAL级别，R4面向HIGH级别，两条规则覆盖同一类证据但在报告中以不同权重呈现，以便分析人员区分高置信度发现与一般性告警。在实现层面，为防止同一注入指令被两条规则重复报告同一证据，系统对发现项以证据字符串为键进行全局去重，确保最终报告的简洁性。

值得注意的是，本文刻意不以"源文件中是否存在整数位运算"作为上述规则的前置门控条件。Triton内核中普遍使用`arith.andi`进行warp内线程编号的位掩码计算（即`tid & 0x1f`），若以"源文件无整数位运算"为门控，此类内核将直接跳过检查，为攻击者提供了通过在源文件中添加一条合法位运算指令绕过检测的途径。可解释集合$\mathcal{E}$已将`0x1f`等常见掩码常量纳入，足以区分合法的warp掩码操作与注入的异常常量，因此无需额外的门控条件。

### X.3.2 存储前序指令序列检测（R2、R7）

R2（PRESTORE\_SEQUENCE）和R7（BROAD\_PRESTORE\_CONST）以SASS中每条全局存储指令（STG/STGX/STL）为锚点，向前扫描若干指令构成的分析窗口，在窗口内寻找与数据篡改相关的特征指令序列。

R2专门检测"浮点计算→异常位运算→全局存储"的三阶段序列。在长度为8的窗口内，R2首先定位最后一条浮点计算指令（FADD、FMUL、FFMA等）的位置$p$，随后在$p$之后、存储指令之前的区间内搜索携带不可解释大常量（$c > \texttt{0x1000}$且$c \notin \mathcal{E}$）的逻辑运算指令。若找到满足条件的指令，则报告CRITICAL级别的发现项，并同时记录该逻辑运算指令和存储指令的原始文本作为证据。R2的检测逻辑要求浮点计算在先、异常位运算在后，这一时序约束显著提高了检测的精确性——正常内核中并非不存在位运算与存储指令出现在同一窗口的情况（例如地址计算中的移位操作），但浮点计算之后紧接以大常量为操作数的位运算，再紧接存储，这一组合在无攻击的情形下概率极低。

R7覆盖R2无法处理的非逻辑运算类注入变体。攻击者可能改用`MOV Rdst, 0x3FE1C71C`将注入值直接写入目标寄存器，或用`IADD3 Rdst, Rdst, 0xXXX, RZ`对计算结果施加整数偏置。这类指令不属于逻辑运算类，R1和R2均无法覆盖。R7使用更小的分析窗口（长度为3，仅取存储指令前最近的3条指令），对窗口内所有非逻辑运算指令检查是否携带不可解释大常量，报告HIGH级别的发现项。较小的窗口减少了将地址计算指令误判为数据篡改指令的概率。

### X.3.3 谓词化数据操作检测（R3）

R3（CONDITIONAL\_DATA\_OP）针对带谓词的逻辑运算指令进行专项检测。攻击者为提高注入的隐蔽性，可能将篡改操作包裹在谓词条件中，使其仅在特定线程或特定输入条件下执行，从而在常规测试中不易触发。R3遍历SASS中所有谓词字段非空（`@P0`、`@!P2`等）的逻辑运算指令，检查其操作数中是否存在满足$c > \texttt{0x1000}$且$c \notin \mathcal{E}$的常量。对于LOP3指令，R3还额外提取操作数中的LUT字节（格式为单字节十六进制数`0xNN`），在发现项中注明具体的逻辑运算类型（例如LUT值`0xfc`对应OR运算），为分析人员提供更丰富的攻击语义信息。

### X.3.4 浮点常量注入检测（R6）

R6（FP\_CONST\_INJECTION）覆盖攻击者以浮点算术运算代替位运算实施注入的变体，例如通过`FMUL R9, R9, 0.888888`将计算结果乘以一个接近1的异常系数，或通过`FADD R9, R9, 0.111111`向结果中叠加固定偏置。此类攻击不涉及任何位运算指令，对规则R1至R4完全透明。

R6在长度为8的窗口内（与R2相同的窗口设置），对助记符属于浮点运算类（FADD、FMUL、FFMA等）的指令，提取操作数中的十进制浮点字面量（格式为`[-]d.ddd`），随后检查该字面量是否属于浮点可解释集合$\mathcal{E}_f$。比较时采用相对容差$\epsilon = 10^{-5}$的近似相等判断，以避免因浮点表示精度差异导致的误报。对参考攻击中的FMUL变体，若攻击者将`or.b32`替换为`FMUL R9, R9, 0.888888`，R6将直接检测到字面量`0.888888`不属于$\mathcal{E}_f$（$\mathcal{E}_f$中包含`0.03125 = 1/32`，但不包含`0.888888`），从而触发HIGH级别的告警。

### X.3.5 全局存储数量异常检测（R8）

R8（STORE\_COUNT\_ANOMALY）从统计层面检测攻击者注入额外全局存储指令（用于向攻击者控制的内存区域写出窃取数据）的行为。R8统计SASS`.text`节区中全局存储类指令（STG、STGX、STL）的数量$N_\text{SASS}$，并从源文件的`tt.store`操作符计数$N_\text{src}$推算合理上限：

$$N_\text{max} = 6 \cdot N_\text{src} + 2$$

系数6是对Triton向量化展开最大倍率的保守估计（一次`tt.store`经向量展开最多对应6条SASS存储指令），常量偏移量2覆盖参数缓冲区写入等额外开销。若$N_\text{SASS} > N_\text{max}$，则触发HIGH级别告警，并在证据字段中注明实际数量与上限，辅助分析人员判断。

## X.4 误报控制

静态分析方法面临的核心挑战之一是误报控制。本章方法在规则设计层面采取了若干系统性措施。

在元数据过滤方面，TTIR文件中的调试位置注解行在预处理阶段被完全剔除，确保路径名中偶然出现的敏感词不会触发R5的关键词扫描。该过滤以严格的格式匹配为条件，不影响正常代码行。MLIR UB方言中的合法标记`ub.poison`被加入关键词白名单，凡包含该标记的行即使与可疑关键词重叠也不触发告警。

在可解释集合的精细化方面，倍数扩展的上界限定为$2^8 = 256$，以避免将过多常量纳入集合而降低灵敏度。常见位掩码的选取依据NVIDIA PTX/SASS汇编器的已知行为，并非启发式枚举。阈值`0x1000`的设定使得编译器正常产生的小整数偏移量不参与检查，在实践中能有效过滤地址计算相关的常量噪声。

在规则结构设计方面，R2要求浮点计算指令在先、异常位运算在后的时序约束，而非仅检查窗口内是否同时存在两类指令，从而将大量地址计算相关的位运算排除在外。R7采用比R2更小的分析窗口（3条vs 8条），进一步限制了将远距离地址操作指令误判为数据篡改指令的可能性。

最后，所有规则的发现项在汇总阶段以证据字符串为键进行去重处理。同一条SASS指令可能同时被R1和R4检测到，去重确保该指令仅以最高严重等级的发现项出现在最终报告中，避免同一注入点的重复计数导致置信度评估失真。

## X.5 本章小结

本章提出了一种基于编译产物一致性的Triton GPU内核注入静态检测方法。该方法以"SASS中出现的常量和操作均应可从源文件合法推导"为基本假设，通过构建可解释常量集合将这一假设转化为可计算的判断准则，并在此基础上设计了八条覆盖不同注入维度的检测规则，涵盖整数位运算注入、浮点算术注入、存储前序数据覆盖写以及额外存储注入等主要攻击变体。该方法无需事先掌握攻击的具体实现，对编译过程中PTX阶段引入的多种恶意变换具有统一的检测能力，且实现轻量、可无缝集成至编译流水线的质量保障环节。

该方法的主要局限体现在三个方面：对于注入常量以寄存器中转而非立即数形式传递的变体，当前方法无法追踪寄存器值的定义来源；对于通过条件跳转将恶意代码隔离在独立基本块中的控制流劫持变体，滑动窗口扫描无法跨越跳转边界；对于通过篡改加载地址使内核从未授权内存区域读取数据的变体，当前方法缺乏对地址计算链的语义分析能力。上述局限需要引入数据流分析和控制流图构建技术加以弥补，这将是下一步研究工作的主要方向。
